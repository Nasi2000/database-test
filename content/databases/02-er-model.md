## ER-диаграмма (Entity-Relationship)

Визуальное представление сущностей предметной области и связей между ними.

## Элементы

- **Сущность** — объект предметной области (Клиент, Заказ)
- **Атрибут** — свойство сущности (email, дата создания)
- **Связь** — отношение между сущностями
- **Кратность** — 1, N, 0..1, 1..*

## Нотация Crow's Foot

```
Клиент ||──o{ Заказ     один клиент — много заказов
Заказ  }o──|| Позиция   один заказ — много позиций
Товар  ||──o{ Позиция   один товар — в многих позициях
```

## Процесс проектирования

1. **Выделить сущности** из требований (существительные: «клиент», «товар»)
2. **Определить атрибуты** для каждой сущности
3. **Найти связи** и кратности
4. **Проверить** на полноту: все сценарии покрыты?
5. **Преобразовать** в логическую схему (таблицы)

## Пример: тикет поддержки

| Сущность | Ключевые атрибуты |
|----------|-------------------|
| Customer | id, email, full_name |
| SupportTicket | id, subject, status, priority |
| Employee | id, full_name, role |

Связи:
- Customer 1:N SupportTicket
- Employee 0..1:N SupportTicket (тикет может быть не назначен)

## Типичные ошибки

- Хранить «список товаров в заказе» одним полем (нужна отдельная таблица)
- Дублировать данные клиента в каждом заказе без причины
- Забыть про слабые сущности (позиция заказа без заказа не существует)

## Практика

Перейдите в **Практика → Проектирование БД** и выполните задание «Система бронирования переговорных».
