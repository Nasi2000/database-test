## Проблема повторной доставки

Сеть ненадёжна. Одно и то же сообщение может прийти **дважды**. Consumer должен обработать это безопасно.

## Идемпотентность

Повторный вызов с теми же данными **не меняет результат**.

**Пример:** `PUT /orders/5/status` с телом `{ "status": "paid" }` — идемпотентно.
**Не идемпотентно:** `POST /orders` без ключа — создаст второй заказ.

## Стратегии

| Стратегия | Как работает |
|-----------|--------------|
| Idempotency-Key | Клиент передаёт уникальный ключ, сервер кэширует результат |
| Natural key | Дедупликация по order_id + event_type |
| Outbox pattern | Событие пишется в БД в той же транзакции, что и бизнес-данные |

## Retry и backoff

```
Попытка 1 → ошибка → ждём 1с
Попытка 2 → ошибка → ждём 2с
Попытка 3 → ошибка → DLQ
```

## Что зафиксировать в спецификации

- Максимум попыток
- Интервалы backoff
- Куда попадают «мёртвые» сообщения
- Кто и как их разбирает (алерт, ручная обработка)
- Гарантия доставки: at-most-once / at-least-once / exactly-once

## Связь с требованиями TechStore

При оплате заказа платёжный шлюз может прислать webhook дважды. Требование:

> Система не должна дважды списать товар или создать дубликат заказа при повторном webhook с тем же payment_id.
