## Зачем аналитику писать тест-кейсы?

Аналитик лучше всех знает **ожидаемое поведение**. Тест-кейсы — мост между требованиями и проверкой.

## Структура тест-кейса

| Поле | Пример |
|------|--------|
| ID | TC-ORD-001 |
| Название | Оформление заказа с валидной корзиной |
| Предусловия | Пользователь авторизован, в корзине 1 товар |
| Шаги | 1. Открыть корзину 2. Нажать «Оформить» 3. Подтвердить |
| Ожидаемый результат | Заказ создан, статус `new`, email отправлен |
| Приоритет | High |

## Позитивные и негативные сценарии

**Позитивный:** всё по правилам → успех.

**Негативный:** нарушение правила → понятная ошибка.

| Сценарий | Тип | Ожидание |
|----------|-----|----------|
| Оформить заказ с товаром в наличии | Позитивный | 201 Created |
| Оформить заказ с quantity = 0 | Негативный | 400 Bad Request |
| Оформить заказ при stock_qty = 0 | Негативный | 409 Conflict |

## Трассировка

```
REQ-042 → US-12 → TC-ORD-001, TC-ORD-002
```

Каждое acceptance criterion должно иметь минимум один тест-кейс.

## Чек-лист хорошего тест-кейса

- [ ] Воспроизводим без догадок
- [ ] Один сценарий — один кейс
- [ ] Ожидаемый результат измерим
- [ ] Есть негативные варианты

## Практика

В разделе **Практика → API** проверьте сценарии создания заказа — это живые негативные и позитивные кейсы.
