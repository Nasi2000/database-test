## User Story

**Формат:** Как `<роль>`, я хочу `<действие>`, чтобы `<ценность>`.

### Пример (TechStore)

> Как **покупатель**, я хочу **видеть историю своих заказов**, чтобы **отслеживать доставку и повторять покупки**.

### Acceptance Criteria (Given / When / Then)

```gherkin
Given покупатель авторизован
  And у него есть минимум 1 заказ
When он открывает раздел "Мои заказы"
Then отображается список заказов с датой, статусом и суммой
  And заказы отсортированы по дате (новые сверху)
```

### INVEST-критерии хорошей story

- **I**ndependent — независима
- **N**egotiable — обсуждаема
- **V**aluable — ценна для пользователя
- **E**stimable — оцениваема
- **S**mall — достаточно мала для спринта
- **T**estable — проверяема

## Use Case

Более формальный сценарий взаимодействия актора с системой.

### Структура

| Поле | Содержание |
|------|-----------|
| Название | Оформить заказ |
| Актор | Покупатель |
| Предусловия | Корзина не пуста, пользователь авторизован |
| Основной поток | 1. Выбрать доставку → 2. Оплатить → 3. Получить подтверждение |
| Альтернативы | Недостаточно товара на складе → показать ошибку |
| Постусловия | Создан заказ со статусом `new` |

## Когда что использовать?

| User Story | Use Case |
|------------|----------|
| Agile / Scrum | Waterfall, формальные проекты |
| Кратко, для бэклога | Детально, с ветвлениями |
| Команда знает контекст | Нужна полная трассировка |
